# Interview Deep-Dive Q&A

## Overview

從 codebase 與 git 歷史中，把「使用者本人做過的某個 feature」整理成面試用的技術深挖 Q&A。核心原則：**每一句回答都要有 code 或 commit 佐證，寧可標記缺口也不要編造**。面試官會追問細節，唬爛的答案比沒有答案更糟。

## 輸入與模式判斷

使用者會給一個 feature 名稱、關鍵字或 issue/ticket 編號。先判斷模式：

- 「整理」「筆記」「summarize」→ **筆記模式**：跑完整工作流程，產出 Q&A 文件
- 「模擬面試」「練習」「考我」「quiz me」→ **模擬面試模式**：見文末該節
- 不明確時預設筆記模式

## 工作流程（筆記模式）

### 1. 定位「使用者本人」的貢獻範圍

面試講的是「你做了什麼」，不是「系統長什麼樣」。先用 `git config user.name` 取得作者名（或問使用者在該 repo 用的 author 名稱），分清楚本人做的 vs 團隊既有的：

```bash
git log --author="$(git config user.name)" --grep="<關鍵字或票號>" -i --oneline --all
git log --author="$(git config user.name)" -i --all --oneline -- '<相關路徑>'   # 用路徑補抓漏網 commit
git show --stat <commit>   # 看每個 commit 動了哪些檔案
```

也用 Grep/Glob 掃關鍵字，找出相關的模組與檔案。若有 GitHub/GitLab/issue tracker 的 MCP 工具或 `gh` CLI，抓 PR/MR 描述與 issue 內容——那裡有需求背景與 review 討論，是「為什麼這樣做」的第一手來源。

### 2. 重建技術全貌

讀 commit 觸及的檔案與其上下游，回答出這幾件事（答不出來就繼續挖，不要跳過）：

- 這個 feature 解決什麼問題？誰會用到？
- 資料流：請求進來 → 經過哪些服務/queue → 存到哪 → 推到哪（依該 repo 的實際架構重建這條鏈）
- 使用者的 code 落在這條鏈的哪幾段？
- 有沒有 fix/refactor 類 commit？那通常是「踩過的坑」，是面試最值錢的素材

### 3. 挖可量化與可追問的點

主動找數字與 trade-off，這是深挖題的核心：

- 效能：commit message 有 "speed up"、"optimize" 就去比對前後做法，估出改善幅度
- 規模：影響幾個 API/多少使用者/多少資料量？從 feature flag、設定檔、資料模型數出來
- 邊界：為什麼用 A 不用 B？失敗時會怎樣？如何相容舊版？

### 4. 產出 Q&A 文件

寫到 `~/interview-notes/<feature-slug>.md`（目錄不存在就建立），全文繁體中文、技術名詞保留英文。結構固定：

```markdown
# <Feature> 面試深挖筆記

## 30 秒版本
（電梯簡介：問題→我的方案→結果，三句話）

## 我做的部分 vs 既有架構
（表格：範圍｜我做的｜佐證 commit/檔案）

## 技術深挖 Q&A
### L1 概念題（這是什麼、為什麼要做）
### L2 設計題（為什麼這樣設計、比較過什麼方案）
### L3 追問題（失敗場景、相容性、效能、規模數字）
### L4 延伸題（如果重來、如果規模 ×10 會改什麼）

每題格式：
**Q:** 面試官可能的問法
**A:** 建議回答（口語、第一人稱，附佐證 `file:line` 或 commit）
**可能追問:** 下一層問題與應對方向

## 誠實邊界
（我沒做的、我不確定的——被問到時如何誠實回答，並帶回自己熟悉的領域）

## 待補資料
（標【需補充】的數字或事實，以及去哪裡查）
```

## 模擬面試模式

扮演資深後端面試官，跟使用者一問一答練習。開始前先建立事實基礎：

1. `~/interview-notes/<feature-slug>.md` 已存在 → 讀它當題庫與評分依據
2. 不存在 → 先執行筆記模式的步驟 1–3（挖 commit、重建全貌），在腦中建立題庫即可，不必先產文件

對話規則：

- **一次只問一題**，問完就停，等使用者回答。絕不自問自答。
- 從 L1 開始，依回答品質調整：答得好 → 往 L2/L3 追問同一主題；答不出 → 給一個提示，再不行就講解並換題。
- 用真實面試官的口吻追問：「為什麼不用 X？」「如果依賴的服務掛了/資料是舊格式會發生什麼事？」「這個改動影響多大範圍？」
- **每題回答後給回饋**再問下一題：
  - 事實正確性：對照 code/commit，答錯的地方直接指出並附佐證
  - 表達：有沒有量化數字、有沒有先講結論、會不會太像背文件
  - 漏講的加分點：哪個 trade-off 或坑沒提到很可惜
- 使用者說「結束」「總結」時收尾：列出強項、弱項、建議回去補的題目清單；若筆記檔存在，把弱項附加到檔案的「待補資料」節。

## 鐵律

- **禁止編造**：找不到佐證的數字/事實，一律寫【需補充：去哪查】，不准填一個看起來合理的值。
- **區分你/團隊**：既有架構（框架、共用基礎設施）可以講，但要標明是團隊基礎設施，本人貢獻在哪一層。
- **答案要口語**：A 欄寫的是「面試時嘴巴講出來的話」，不是文件語氣。
- 產出檔案放 `~/interview-notes/`，**絕不寫進公司 repo**。

## 常見錯誤

| 錯誤 | 修正 |
|------|------|
| 只 grep 現有 code，沒看 git 歷史 | 現有 code 看不出「誰做的、為何這樣改」，必須配 `git log --author` |
| 把整個系統架構當成本人成果 | 每個主張對到本人的 commit |
| Q&A 只有 happy path | L3 一定要有失敗場景、相容性、rollback 題 |
| 用文件語氣寫答案 | 改寫成第一人稱口語，能自然講出來 |
| 模擬面試時一次列出一整串題目 | 一次一題，等回答、給回饋，才有練習效果 |
| 模擬面試憑印象評分 | 回饋必須對照 code/commit 事實，答錯要附佐證 |
